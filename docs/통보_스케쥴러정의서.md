# PreCheck 통보 스케쥴정의서 v1.0

## 문서의 목적
- 통보 시스템에서 참고할 통보 실행 시각/주기를 정의한 파일을 설명
- 통보 시스템내 파일 형태로 정의함
- 수집/분석 스케쥴 정의서와 달리, 통보 스케쥴은 특정 서버나 파일을 대상으로 지정하지 않음 — 스케쥴이 도래하면 그 시점에 `TB_ANALYZE_RESULT`에 존재하는 모든 `SERVER_ID`를 한 번에 일괄 검사함

---

## 통보 정의서 파일명
- 통보 시스템 내 ~/cfg/PreCheck_NotifyLogs_Schedule.conf (collect/analyze 스케쥴 파일과 동일한 위치 컨벤션)

---

## 통보 주기
- 통보 주기는 다음과 같음
- 배치 -> 지정 시각에 1회, 등록된 모든 서버를 대상으로 집계/통보 점검 실행
- 주기 -> 정해진 시간부터 주기별로, 등록된 모든 서버를 대상으로 집계/통보 점검 실행

---

## 통보 정의 파일 포맷
- 수집/분석 정의 파일과 달리 **서버구분, 대상파일명 필드가 없음** — 통보주기기술만 한 줄에 기술함
- [통보주기기술]

- 통보주기기술은 통보구분별로 다음의 포맷으로 정의 (수집/분석 정의서와 동일한 표기 규칙 사용)
    -> 배치 : [배치|요일|시작시간]
    -> 주기 : [주기|요일|시작시간|통보간격(분)|종료시간]
        -> 요일 표기 규칙
            * 단일 요일 : 0~6 중 하나 (0->일요일, 1->월요일, 2->화요일, ..., 6->토요일)
            * 범위      : 시작-끝 형식 (예: 1-5 = 월~금)
                          - 시작은 끝보다 작거나 같아야 함
                          - 0-6은 *과 동일 (매일)
            * 전체      : * (매일)
            * 콤마(,) 또는 복합 표기(예: 1,3,5 또는 1-3,5)는 1차 버전에서 미지원
            * 위 규칙을 따르지 않는 요일 표기는 스케쥴 무시 처리
        -> 시작시간 : HHmmss, 시작시간은 000001이 최소임
        -> 통보간격(분) : 분단위(예 30:30분)
        -> 종료시간 : HHmmss, 종료시간은 235959가 최대임

- 파일은 텍스트 형태며 개행문자(\n)로 구분
- 위 포맷을 따르지 않는 스케쥴은 무시함
- skip처리는 #으로 시작함
- 동일한 통보주기기술이 중복 등록되어도 동작은 같으나(같은 시각에 같은 점검을 두 번 실행), 운영 편의상 중복 등록은 지양함

## 통보 스케쥴 동작 방식

```
[통보 스케쥴 실행 시각 도래]
        |
        ↓
TB_ANALYZE_RESULT에 존재하는 SERVER_ID 전체 목록 조회
        |
        ↓
서버별로 순회:
   ① TB_NOTIFY_HISTORY에서 이 서버의 마지막 SUCCESS/PARTIAL 행의 AGGREGATE_TO 조회 (워터마크)
      → 없으면(신규 서버) AGGREGATE_FROM 기본값 사용
         - 주기 스케쥴: 이번 실행 시각 - 1주기(통보간격분)
         - 배치 스케쥴: 당일 00:00:00
   ② AGGREGATE_FROM(워터마크) ~ 이번 스케쥴 슬롯 시각 구간으로 정상/경고/에러 건수 집계
      ※ "이번 스케쥴 슬롯 시각"은 스케쥴러가 실제로 감지한 wall-clock이 아니라
        conf에 정의된 슬롯 시각(예: 09:00:01) 그대로 사용함 — 폴링 지연이 누적돼도
        구간 경계가 흔들리지 않도록 함 (collect/analyze 폴링 패턴과 동일 원칙)
   ③ 경고+에러 합계 = 0 → 이 서버는 스킵 (TB_NOTIFY_HISTORY 행 생성 안함)
      경고+에러 합계 ≥ 1 → 통보 대상에 추가
        |
        ↓
통보 대상 서버가 1개 이상이면, TCP connection 1개 open (이번 런의 모든 서버가 이 connection 하나를 공유)
   → bind(01) 1회
      - bind 실패 → 통보 대상 서버 전원 FAIL(0건 시도)로 즉시 기록, connection close 후 런 종료
   → 통보 대상 서버를 한 서버씩 순서대로 처리:
        해당 서버 × 전송대상 파일의 수신자 전화번호 조합마다 submit(03) 반복
        (전송대상 파일이 비어있으면 submit 0건으로 SUCCESS 처리, 7절 참고)
        → 그 서버 처리가 끝나는 즉시 TB_NOTIFY_HISTORY INSERT 후 다음 서버로 진행 (crash-safe, 일괄 INSERT 아님)
        → 처리 중 connection이 끊기면 그 서버는 PARTIAL/FAIL로 즉시 기록하고,
          아직 처리 안 한 나머지 서버는 전부 FAIL(0건 시도)로 기록 후 런 종료(재연결 안 함)
   → 모든 서버 처리(또는 중단 처리) 완료 후 connection close
        |
        ↓
각 서버는 INSERT 직후 해당 NOTIFY_STATUS에 따라 TB_ANALYZE_RESULT.NOTIFY_YN 갱신 여부 결정
(상세는 통보_DB정의서.md, 통보_TR연동정의서.md 참고)
```

## 파일 공통 규칙
- 파일은 텍스트 형태이며 개행문자(\n)로 구분
- 위 포맷을 따르지 않는 스케쥴은 무시함
- skip 처리는 #으로 시작함
- 통보 실패 시 재시도 정책은 **수집/분석과 다름**:
  - 수집/분석: 최초 시도 실패 후 10초 간격 3회 즉시 재시도 (프로그램 명세서 공통 정책)
  - 통보: 즉시 재시도 없음. 서버별로 connect/bind 자체가 실패(FAIL)하면 그 서버의 워터마크를 전진시키지 않고, **다음 통보 스케쥴 실행 시각에 동일 구간을 재집계**하는 방식으로 재시도를 대체함. 일부 수신자만 전송 실패(PARTIAL)한 경우는 재시도하지 않고 다음 구간으로 진행함

## 수신자(전송대상) 파일 규칙

- 파일명: `PreCheck_NotifyTarget_List.conf` (통보 스케쥴 conf 파일과 동일한 위치 컨벤션, 예: `~/cfg/PreCheck_NotifyTarget_List.conf`)
- 인코딩: UTF-8 (TR 전송 시 사용하는 EUC-KR과는 별개 — 이 파일은 통보 서버가 직접 읽는 설정 파일이므로 다른 conf 파일들과 동일하게 UTF-8 사용)
- 형식: 한 줄에 전화번호 1개, 이름 등 추가 컬럼 없음 (전역 공통 목록, 서버/LOG_ID별 구분 없음)
- 파싱 규칙
  - `#`으로 시작하는 줄은 skip 처리
  - 빈 줄은 무시
  - 숫자가 아닌 문자(하이픈 등)는 제거 후 숫자만 추출 (예: `010-1234-5678` → `01012345678`)
  - 추출 후 동일한 번호가 중복되면 1개로 합침 (같은 번호로 SMS가 중복 발송되지 않도록)
- 파일이 비어있거나(유효한 번호 0개) 존재하지 않으면, 통보 대상 서버가 있어도 submit을 0건 시도하고 `NOTIFY_STATUS='SUCCESS'`로 처리함(`RECV_TOTAL_COUNT=0`)
  - 즉, 이 파일을 비워두는 것은 "통보를 의도적으로 끄는" 운영 수단으로 사용할 수 있음
  - bind 자체는 시도하지 않음(보낼 대상이 없으므로 connection도 열지 않음) — 단, 이 경우에도 워터마크는 전진시킴

## 중복 실행 방지

- collect/analyze 스케쥴러는 `서버구분 + 경로 + 스케쥴식`을 조합한 키로, 같은 폴링 윈도우 안에서 동일 스케쥴이 두 번 실행되지 않도록 추적한다 (`buildScheduleKey`)
- 통보 스케쥴은 서버/경로 구분이 없는 전역 한 줄이므로, 키를 **"통보주기기술 문자열 자체 + 매칭된 슬롯 시각"** 조합으로 만들어 동일한 방식으로 추적한다
  - 예) `[배치|1-5|180001]` 스케쥴이 2026-06-18 18:00:01 슬롯에 매칭됐다면, 키는 `"배치|1-5|180001" + "20260618180001"` 형태가 됨
  - 같은 키가 폴링 윈도우(`pollWindowSeconds`) 안에서 다시 매칭되면 중복 실행으로 간주해 무시함

---

## 스케쥴 예
- `#[배치|*|090001]` -> skip
- `[배치|1-5|180001]` -> 월~금요일 18시 00분 01초에 전체 서버 대상 통보 점검을 1회 실행
- `[배치|*|090001]` -> 매일 9시 00분 01초에 전체 서버 대상 통보 점검을 1회 실행
- `[주기|*|090001|30|180001]` -> 매일 9시 00분 01초부터 30분 간격으로 18시 00분 01초까지 전체 서버 대상 통보 점검을 반복 실행
- `[주기|1-5|083001|10|173001]` -> 월~금요일 8시 30분 01초부터 10분 간격으로 17시 30분 01초까지 전체 서버 대상 통보 점검을 반복 실행

## 수집/분석 스케쥴 정의서와의 차이점 요약

| 항목 | 수집 스케쥴 정의서 | 분석 스케쥴 정의서 | 통보 스케쥴 정의서 |
|---|---|---|---|
| 대상 식별 | 서버구분 + IP주소 + 파일경로 | 서버구분 + (파일경로) | **없음 (전역, 매번 전체 서버 검사)** |
| 대상 지정 방식 | 파일 단위만 가능 | 서버 단위 또는 파일 단위 | 대상 지정 불가, 시각만 정의 |
| 중복 방지 수단 | TB_COLLECT_HISTORY.LAST_LINE_NUMBER | TB_ANALYZE_HISTORY.LAST_ANALYZE_LOG_ID | TB_NOTIFY_HISTORY.AGGREGATE_TO (서버별 워터마크) |
| 실패 재시도 정책 | 10초 간격 3회 즉시 재시도 | 10초 간격 3회 즉시 재시도 | **즉시 재시도 없음, 다음 스케쥴로 위임** |
| 연결 방식 | SFTP, 매 수집마다 새 연결 | DB 조회만 (외부 연결 없음) | TCP, 스케쥴 실행마다 새 connection (비연결형) |
